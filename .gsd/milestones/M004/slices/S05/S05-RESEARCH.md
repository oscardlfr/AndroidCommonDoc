# Phase 14.3: Skill Materialization & Registry - Research

**Researched:** 2026-03-15
**Domain:** File distribution, content hashing, manifest-driven sync, docs naming normalization, CLAUDE.md layering
**Confidence:** HIGH

## Summary

Phase 14.3 replaces the current fragile path-based delegate pattern with a professional registry + manifest + materialization system. The current state has 27 L0 skills, 11 L0 agents, and 32 L0 commands distributed to downstream projects (shared-kmp-libs: 9 delegate commands; DawSync: 29 delegate commands, 6 delegate agents, 6 junction-based delegate skills) via `install-claude-skills.sh` and manual delegate stubs with `delegate: true` + `canonical: path` frontmatter. This approach is brittle -- delegates reference paths that can break, junctions are Windows-specific, and there is no version tracking or drift detection.

The solution architecture is straightforward: a `skills/registry.json` auto-generated from filesystem scanning catalogs all L0 assets with SHA-256 content hashes; each downstream project declares an `l0-manifest.json` specifying which L0 assets to adopt; a `/sync-l0` skill materializes full copies with version headers, replacing all delegate stubs, junctions, and the `install-claude-skills.sh` script entirely. Additionally, ~84 active UPPERCASE DawSync docs need renaming to `lowercase-kebab-case`, 23 category tags need consolidation to ~8, CLAUDE.md files need de-duplication across the L0/L1/L2 hierarchy, and the Claude Code adapter pipeline should be simplified by having SKILL.md serve directly without intermediate command generation.

**Primary recommendation:** Build the registry and sync engine as a Node.js TypeScript tool within the existing MCP server codebase (leveraging the scanner, frontmatter parser, and project-discovery infrastructure already there), then execute a big-bang migration converting all delegates in one pass.

### Pattern 2: Manifest Schema (l0-manifest.json)
**What:** Per-project declaration of which L0 assets to adopt
**When to use:** Every L1/L2 project that consumes L0 skills
**Recommendation (Claude's Discretion): include-all with excludes.**
Rationale: Solo developer workflow optimized for speed. New L0 skills auto-propagate to all downstream projects. Explicit excludes only when a skill is irrelevant (e.g., web skills excluded from shared-kmp-libs). ### Pattern 3: Materialized Copy Format
**What:** Full content copy with inline version header instead of delegate stub
**Recommendation (Claude's Discretion): Inline YAML frontmatter extension.**
Rationale: Sidecar files add clutter and can get out of sync. Inline header is self-documenting, visible to Claude Code, and requires no tooling to read. The version info is injected into the existing YAML frontmatter block.
**Example (materialized skill):**

```markdown
l0_hash: "sha256:abc123def456..."
l0_synced: "2026-03-15T12:00:00Z"
---

## Usage Examples
...
```

**Example (materialized command):**
```markdown
<!-- L0 Materialized | source: AndroidCommonDoc/.claude/commands/test.md | hash: sha256:abc123... | synced: 2026-03-15 -->
# /test - Run Tests with Smart Retry
...
```

### Pattern 4: Adapter Pipeline Simplification
**What:** Claude Code reads SKILL.md directly; eliminate intermediate command generation for Claude Code
**Current flow:** `SKILL.md -> claude-adapter.sh -> .claude/commands/{name}.md -> Claude Code reads command`
**Simplified flow:** `SKILL.md -> Claude Code reads skill directly` (Claude Code already discovers `skills/*/SKILL.md` natively)
**Impact:** The `claude-adapter.sh` script and generated commands become redundant for Claude Code. Commands that are NOT backed by skills (16 standalone commands like `brainstorm`, `bump-version`, etc.) remain as-is in `.claude/commands/`. The `copilot-adapter.sh` is retained for Copilot prompt generation.
**Key insight:** Claude Code already discovers and uses `skills/*/SKILL.md` files directly. The 16 generated command files in `.claude/commands/` that duplicate skill content can be removed from L0. Downstream projects that get materialized copies of skills no longer need the command duplicates either.

### Pattern 5: CLAUDE.md Layering
**What:** De-duplicate CLAUDE.md across L0/L1/L2 using convention-based references
**Current state:**
- `~/.claude/CLAUDE.md` (101 lines) = Full KMP rules (this IS the shared base)
- `AndroidCommonDoc/CLAUDE.md` (101 lines) = Exact duplicate of ~/.claude/CLAUDE.md
- `shared-kmp-libs/CLAUDE.md` (57 lines) = Module-specific, some KMP rule duplication
- `DawSync/CLAUDE.md` (245 lines) = Project-specific + massive KMP rule duplication

**Recommendation (Claude's Discretion): Convention-based slimming.**
- `~/.claude/CLAUDE.md` stays as-is (101 lines) -- Claude Code auto-loads this for ALL projects
- `AndroidCommonDoc/CLAUDE.md` becomes toolkit-specific only (no KMP rules duplication since ~/.claude/ already provides them)
- `shared-kmp-libs/CLAUDE.md` stays module-focused (57 lines) -- minimal duplication
- `DawSync/CLAUDE.md` slimmed to project-specific rules only (domain architecture, module structure, wave tracking) -- KMP rules come from ~/.claude/CLAUDE.md. Target: <120 lines

**Key insight:** Claude Code loads `~/.claude/CLAUDE.md` + `{project}/CLAUDE.md` automatically. There is no need for `@import` or explicit inheritance. Simply remove duplicated content from project CLAUDE.md files, keeping only project-specific rules. The user's context budget constraint is <150 lines per file, <4000 tokens.

### Anti-Patterns to Avoid
- **Generating delegates instead of full copies:** Delegates break when paths change, have no version tracking, and require the L0 repo to be accessible at runtime
- **Manual version bumps:** Content hashes are deterministic and automatic -- never require human intervention
- **Interactive sync prompts:** User decided on `git pull` semantics -- show diff, auto-apply, no confirmation
- **Splitting /sync-l0 across multiple tools:** Keep it as one cohesive skill that handles the full workflow (scan registry, read manifest, compare hashes, materialize, update checksums)

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SHA-256 hashing | Custom hash function | `node:crypto.createHash('sha256')` | Battle-tested, built-in, correct |
| Frontmatter parsing | Custom YAML parser | Existing `mcp-server/src/registry/frontmatter.ts` | Already handles all edge cases, tested |
| Filesystem scanning | Custom walker | Existing `mcp-server/src/registry/scanner.ts` pattern | Already handles archive exclusion, recursive dirs |
| Project discovery | Hardcoded paths | Existing `project-discovery.ts` | Already discovers from settings.gradle.kts and vault-config.json |
| Cross-reference updates | Manual find/replace | Scripted regex with frontmatter-aware parsing | Avoid breaking YAML blocks, code blocks, wikilinks |
| File rename with git tracking | `mv` + `git add` | `git mv` | Preserves git history, handles staging |

**Key insight:** The MCP server codebase already has 80% of the infrastructure needed. The scanner, frontmatter parser, project-discovery, and validation patterns are all proven and tested (422 tests). Extend them rather than building parallel systems.

## Common Pitfalls

### Pitfall 1: Windows Junction Cleanup
**What goes wrong:** DawSync `.claude/skills/` contains 6 Windows NTFS junctions pointing to `.agents/skills/`. Deleting junction targets without removing junctions first causes dangling pointers.
**Why it happens:** Junctions are not regular directories -- `rm -rf` on the junction removes the target data, not the junction itself.
**How to avoid:** Remove junctions first with `cmd /c "rmdir .claude\\skills\\accessibility"` (Windows) before recreating as regular directories with materialized copies.
**Warning signs:** `<JUNCTION>` in `dir` output; permission errors when trying to write to junction paths.

### Pitfall 2: DawSync Doc Rename Cross-Reference Breakage
**What goes wrong:** Renaming ~146 active docs from UPPERCASE to lowercase-kebab-case breaks wikilinks, `parent:` frontmatter fields, hub `## Sub-documents` sections, l0_refs references, and any hardcoded paths in CLAUDE.md or other files.
**Why it happens:** The codebase has extensive cross-referencing: 51 sub-docs with `parent:` fields (Phase 14.2.1-04 already fixed case sensitivity), vault wikilinks, MOC pages, and README.md references.
**How to avoid:** Build a rename map (old -> new) first, then script the bulk rename with cross-reference updates in a single pass. Use `git mv` for history preservation. Run validate-doc-structure after to catch any missed references.
**Warning signs:** Broken wikilinks in Obsidian, orphaned sub-docs (parent not found), MCP validation errors.

### Pitfall 3: Forgetting L2-Specific Files During Migration
**What goes wrong:** The sync tool overwrites or deletes L2-specific commands/agents that should be preserved.
**Why it happens:** DawSync has 10 L2-specific commands (deploy-web, lint-web, test-m4l, nuke-builds, roadmap, run-clean, start-track, sync-roadmap, validate-strings, merge-track) and 5 L2-specific agents (daw-guardian, data-layer-specialist, domain-model-specialist, freemium-gate-checker, producer-consumer-validator) that must NOT be touched by `/sync-l0`.
**How to avoid:** The manifest's `l2_specific` section explicitly lists files that belong to the project. `/sync-l0` must never touch files not in the manifest's checksum list.
**Warning signs:** L2-specific commands missing after sync; `l0_source` header appearing in domain-specific files.

### Pitfall 4: Circular Adapter Dependency
**What goes wrong:** Removing claude-adapter.sh-generated commands from L0 breaks downstream projects that still reference command paths in their delegates.
**Why it happens:** During migration, the old delegate pattern references `.claude/commands/test.md` which was generated by the adapter. If we remove the generated commands before materializing the new copies, there's a gap.
**How to avoid:** Materialize new copies BEFORE deleting old infrastructure. The migration sequence must be: (1) create registry, (2) create manifests, (3) run /sync-l0 to materialize, (4) verify everything works, (5) THEN delete old delegate stubs and install scripts.
**Warning signs:** Claude Code not finding expected commands; skill-script-alignment validator reporting broken references.

### Pitfall 5: Registry Entry Name Collisions
**What goes wrong:** A skill named "test" and a command named "test" collide in the registry.
**Why it happens:** Skills, agents, and commands share a namespace from the user's perspective (both invoked via `/test`).
**How to avoid:** Registry entries include `type` field (skill/agent/command). The `name+type` combination is the unique key, not `name` alone. The manifest also selects by type. For commands generated from skills, the registry marks `generated_from` to show the relationship.
**Warning signs:** Duplicate entries in registry; manifest selecting wrong type.

### Pitfall 6: CLAUDE.md Over-Pruning
**What goes wrong:** Removing "duplicated" rules from project CLAUDE.md that are actually project-specific refinements, not pure duplicates.
**Why it happens:** DawSync CLAUDE.md has rules that look identical to ~/.claude/CLAUDE.md but include DawSync-specific context (e.g., "Tests run sequentially" is in both, but DawSync adds the Windows file-locking rationale).
**How to avoid:** Diff each rule carefully. Keep rules that add project-specific detail, remove only pure duplicates. The rule: if ~/.claude/CLAUDE.md says it and the project CLAUDE.md adds nothing, remove it. If it adds context, keep a shortened version with the context.
**Warning signs:** Agent losing awareness of project-specific testing constraints or architecture patterns after CLAUDE.md slimming.

### Registry Generation (extend scanner.ts pattern)
```typescript
// Source: Extends existing mcp-server/src/registry/scanner.ts pattern
import { createHash } from "node:crypto";
import { readFile, readdir } from "node:fs/promises";
import path from "node:path";
import { parseFrontmatter } from "./frontmatter.js";

interface RegistryEntry {
  name: string;
  type: "skill" | "agent" | "command";
  path: string;
  description: string;
  category: string;
  tier: "core" | "extended" | "web";
  hash: string;
  dependencies: string[];
  generated_from?: string;
  frontmatter: Record<string, unknown>;
}

async function computeHash(filePath: string): Promise<string> {
  const content = await readFile(filePath);
  const hash = createHash("sha256").update(content).digest("hex");
  return `sha256:${hash}`;
}

async function scanSkills(rootDir: string): Promise<RegistryEntry[]> {
  const skillsDir = path.join(rootDir, "skills");
  const entries: RegistryEntry[] = [];
  const dirs = await readdir(skillsDir, { withFileTypes: true });

  for (const dir of dirs) {
    if (!dir.isDirectory()) continue;
    const skillMd = path.join(skillsDir, dir.name, "SKILL.md");
    try {
      const content = await readFile(skillMd, "utf-8");
      const fm = parseFrontmatter(content);
      entries.push({
        name: dir.name,
        type: "skill",
        path: `skills/${dir.name}/SKILL.md`,
        description: fm.metadata?.description ?? "",
        category: categorize(dir.name),
        tier: tierize(dir.name),
        hash: await computeHash(skillMd),
        dependencies: extractDependencies(content),
        frontmatter: fm.metadata ?? {},
      });
    } catch { /* skip non-skill dirs */ }
  }
  return entries;
}
```

### Materialization with Version Header
```typescript
// Source: New mcp-server/src/sync/sync-engine.ts pattern
function materializeSkill(
  originalContent: string,
  sourcePath: string,
  hash: string,
): string {
  // Inject l0_source, l0_hash, l0_synced into existing YAML frontmatter
  const now = new Date().toISOString();
  const lines = originalContent.split("\n");
  const fmEnd = lines.indexOf("---", 1); // second ---

  if (fmEnd > 0) {
    // Insert before closing ---
    lines.splice(fmEnd, 0,
      `l0_source: "${sourcePath}"`,
      `l0_hash: "${hash}"`,
      `l0_synced: "${now}"`,
    );
  }
  return lines.join("\n");
}

function materializeCommand(
  originalContent: string,
  sourcePath: string,
  hash: string,
): string {
  // Commands may not have YAML frontmatter; use HTML comment header
  const now = new Date().toISOString().split("T")[0];
  const header = `<!-- L0 Materialized | source: ${sourcePath} | hash: ${hash} | synced: ${now} -->\n`;
  // Remove any existing GENERATED comment
  const cleaned = originalContent.replace(/^<!-- GENERATED.*-->\n?<!-- Regenerate.*-->\n?/m, "");
  return header + cleaned;
}
```

### Bulk File Rename Script Pattern
```typescript
// Source: Pattern for MATL-08 DawSync doc rename
interface RenameEntry {
  oldPath: string;
  newPath: string;
  oldSlug: string;
  newSlug: string;
}

function toKebabCase(filename: string): string {
  return filename
    .replace(/\.md$/, "")
    .replace(/_/g, "-")
    .toLowerCase()
    + ".md";
}

// Cross-reference update: frontmatter parent, wikilinks, hub Sub-documents
function updateCrossReferences(
  content: string,
  renameMap: Map<string, string>,
): string {
  let updated = content;
  for (const [oldName, newName] of renameMap) {
    const oldBase = oldName.replace(".md", "");
    const newBase = newName.replace(".md", "");
    // Update parent: field
    updated = updated.replace(
      new RegExp(`^(parent:\\s*)${escapeRegex(oldBase)}`, "m"),
      `$1${newBase}`,
    );
    // Update wikilinks [[OLD_NAME]] -> [[new-name]]
    updated = updated.replace(
      new RegExp(`\\[\\[${escapeRegex(oldBase)}\\]\\]`, "g"),
      `[[${newBase}]]`,
    );
    // Update relative paths
    updated = updated.replace(
      new RegExp(escapeRegex(oldName), "g"),
      newName,
    );
  }
  return updated;
}
```

## Open Questions

1. **Should adapter-generated commands be deleted from L0?**
   - What we know: Claude Code discovers `skills/*/SKILL.md` directly. The 16 generated `.claude/commands/*.md` files that duplicate skill content are redundant for Claude Code users.
   - What's unclear: Whether removing them from L0 breaks any workflow. Copilot adapter reads from SKILL.md directly (not from .claude/commands/), so Copilot is fine.
   - Recommendation: Delete the 16 adapter-generated commands from L0 `.claude/commands/` since they are redundant. Keep the 16 standalone commands (brainstorm, bump-version, etc.) that have no backing skill. Downstream projects get materialized SKILL.md files directly.

2. **Diagram file naming convention**
   - What we know: 62 diagram files use `A01-system-overview.md` format with section-prefix numbering. The prefix codes (A-H) serve as grouping.
   - What's unclear: Whether `A01-` prefixes violate "no numeric prefixes" or if they serve a valid organizational purpose within the diagrams/ hierarchy.
   - Recommendation: Keep the A-H section prefix (it provides meaningful grouping within diagrams) but convert to lowercase-kebab-case: `A01-system-overview.md` stays as-is (it's already lowercase-kebab-case with a valid prefix). The uppercase is in the parent directory names like `A-system-global/` which should become `a-system-global/`.

3. **Should /sync-l0 be a SKILL.md or a standalone script?**
   - What we know: It needs to be invokable from any downstream project. Skills are discovered per-project.
   - What's unclear: A skill in L0 would need to be materialized to downstream projects before it can run there -- chicken-and-egg.
   - Recommendation: Build as a standalone Node.js script (`scripts/sync-l0.cjs` or similar) that can be run directly via `node path/to/sync-l0.cjs`. Also create a SKILL.md wrapper for L0 use. First-time setup in a downstream project uses the direct script path.
