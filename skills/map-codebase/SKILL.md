# Map Codebase Skill

Structured codebase analysis using the `codebase-mapper` agent.

## Usage

```
/map-codebase [mode] [focus]
```

**Modes:**
- `--inventory` (default): Generate MODULE_MAP.md at project root — quick module reference
- `--full [focus]`: Deep analysis by focus area (tech/arch/quality/concerns)

## Steps

### Mode: --inventory (default)

1. Spawn the codebase-mapper agent with inventory prompt:

```
Agent(
  subagent_type="codebase-mapper",
  prompt="Generate a MODULE_MAP.md for this project at {cwd}.

Scan all modules (build.gradle.kts files) and for each module:
- Module name (Gradle path)
- What it provides (1 sentence)
- Key public APIs (top 3-5 classes/interfaces/functions)
- Source set (commonMain/jvmMain/androidMain/etc.)

Write MODULE_MAP.md to the project root with this format:

# Module Map
> Auto-generated by /map-codebase. Re-run to refresh.

| Module | Provides | Key APIs | Source Set |
|--------|----------|----------|------------|

## Decision Tree
- Need X? → module-name
- Need Y? → module-name

Keep it under 100 lines. Focus on REUSABLE modules, skip app/test modules.",
  description="Generate MODULE_MAP.md"
)
```

2. Verify MODULE_MAP.md was created at project root

### Mode: --full [focus]

1. Parse focus area from arguments (default to `arch`)
2. Focus areas: `tech`, `arch`, `quality`, `concerns`
3. Spawn the codebase-mapper agent:

```
Agent(
  subagent_type="codebase-mapper",
  prompt="Analyze this codebase with focus: {focus}\n\nProject root: {cwd}\nWrite a structured analysis document.",
  description="Map codebase: {focus}"
)
```

4. Present the analysis to the user

## Notes

- Default mode is `--inventory` — designed for plan mode context
- MODULE_MAP.md should be committed to the repo (it's documentation, not generated code)
- Re-run when modules are added/removed/renamed
- Boris Cherny principle: CLAUDE.md = workflow rules, MODULE_MAP.md = what exists
