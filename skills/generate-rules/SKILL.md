## Usage Examples

```
/generate-rules
/generate-rules --dry-run
/generate-rules --consumer path/to/project
/generate-rules --consumer ../shared-kmp-libs --rules-dir detekt-rules-l1 --package com.grinx.shared.detekt.rules.generated
```

## Parameters

Uses parameters from `params.json`:
- `project-root` -- Path to the AndroidCommonDoc toolkit root directory.

Additional skill-specific arguments (not in params.json):
- `--dry-run` -- Preview which rules would be generated, skipped, or removed without writing files (default behavior for safety).
- `--consumer path/to/project` -- Generate rules from pattern docs in a consumer project's `docs/` directory.
- `--rules-dir module-name` -- Detekt rules module directory (default: `detekt-rules`). Use `detekt-rules-l1` for L1, etc.
- `--package com.example.detekt.rules.generated` -- Kotlin package for generated rule classes. Must match the module's source tree.

## Behavior

1. Run the `generate-detekt-rules` MCP tool in dry-run mode first to preview changes.
2. Show the user a summary of planned actions:
   - **Generate** -- New rules to create (rule ID, type, source pattern doc).
   - **Skip** -- Rules marked `hand_written: true` that will not be overwritten.
   - **Remove** -- Orphaned generated rules no longer defined in frontmatter.
3. On user confirmation: re-run the tool without dry-run to write the Kotlin source files.
4. Run `./gradlew :detekt-rules:test` to verify the generated rules compile and their tests pass.
5. If new rules were generated: display the `providerUpdateBlock` and remind the user to update `AndroidCommonDocRuleSetProvider.kt` with the new rule registrations.

## Implementation

This skill is an orchestration workflow using the AI agent's built-in tools.

The agent performs the following steps:
1. Call the `generate-detekt-rules` MCP tool with `dry_run: true`.
2. Parse the `GenerationResult` JSON response.
3. Display a preview table: generated rules, skipped hand-written rules, and orphaned removals.
4. Ask the user for confirmation to proceed.
5. On confirmation: call `generate-detekt-rules` with `dry_run: false`.
6. Use `Bash` to run `./gradlew :detekt-rules:test` and verify compilation.
7. If new rules added: show the provider update block for `AndroidCommonDocRuleSetProvider.kt`.

## L0/L1 Config Hierarchy

AndroidCommonDoc ships two Detekt configs:

- `detekt-l0-base.yml` — **baseline distributed with the toolkit**. All rules `active: true`. Generated by `emitBaselineConfig()` from frontmatter — never edit by hand.
- `config.yml` — example L1 override. L1 projects only declare what they want to change.

After running `/generate-rules`, the `detekt-l0-base.yml` is regenerated to include any new rules with `active: true`.

L1 consumer project usage:
```bash
./gradlew detekt --config ~/.androidcommondoc/detekt-l0-base.yml,detekt.yml
```

Detekt merges with `CompositeConfig(lookFirst=detekt.yml, lookSecond=base.yml)` — the L1 file wins per leaf key.

See `docs/guides/detekt-config.md` for the full rule catalog and override patterns.
