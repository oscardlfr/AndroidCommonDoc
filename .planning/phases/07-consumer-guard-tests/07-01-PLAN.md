Create `guard-templates/` directory with 6 template files. All templates use `__ROOT_PACKAGE__` as the token for sed substitution. All package declarations use `__ROOT_PACKAGE__.konsist.guard`.

**build.gradle.kts.template:**
- `kotlin("jvm")` WITHOUT version (inherits from consumer's pluginManagement -- per research recommendation, avoids "Kotlin plugin applied multiple times" conflict)
- Pinned library versions: `konsist:0.17.3`, `junit-jupiter:5.11.4`, `assertj-core:3.27.3`, `junit-platform-launcher` (runtime)
- `useJUnitPlatform()`, `outputs.upToDateWhen { false }`, `maxParallelForks = 1`
- `group = "__ROOT_PACKAGE__"` (substituted by install script)

**GuardScopeFactory.kt.template:**
- `object GuardScopeFactory` with `private const val ROOT = "__ROOT_PACKAGE__"`
- `projectScope()`: calls `Konsist.scopeFromProduction()`, filters by ROOT package, `require(count > 0)` canary with actionable message including working dir
- `hasFilesInPackage(scope, packagePrefix)`: returns Boolean for layer pre-checks
- KDoc explaining this is generated from AndroidCommonDoc guard-templates

**ArchitectureGuardTest.kt.template:**
- Uses `GuardScopeFactory.projectScope()` for scope
- Defines layers with `__ROOT_PACKAGE__` prefix: Model (`core.model..`), Domain (`core.domain..`), Data (`core.data..`), Feature (`feature..`)
- Each test uses `assumeTrue(GuardScopeFactory.hasFilesInPackage(...))` to skip if layer has no files
- Test 1: `model layer does not depend on other layers` -- import-based check on model files for domain/data/feature imports
- Test 2: `domain layer does not depend on data or feature layers` -- import-based check on domain files
- Test 3: `data layer does not depend on feature layer` -- import-based check on data files
- Use AssertJ `withFailMessage` for violations (names offending file + import + remediation)
- NOTE: Use import-based checks (not Konsist assertArchitecture with Layer objects) to avoid KoPreconditionFailedException when layers are empty. The assumeTrue check guards whether we enter the test at all, and the import-based assertions within each test are straightforward. This is the safest approach for consumers with varying project structures.

**NamingGuardTest.kt.template:**
- Uses `GuardScopeFactory.projectScope()` for scope
- Test 1: `classes in domain usecase package end with UseCase` -- `withPackage("$root..usecase..")`, `assumeTrue` if no usecase classes found
- Test 2: `classes ending with ViewModel reside in feature package` -- `withNameEndingWith("ViewModel")`, `assumeTrue` if none, AssertJ assertion on `resideInPackage("$root.feature..")`
- Test 3: `classes ending with Repository reside in data or domain package` -- `withNameEndingWith("Repository")`, verify package is `core.data..` or `core.domain..`
- Generic guidance messages (not pattern doc links -- per discretion recommendation to avoid coupling)

**PackageGuardTest.kt.template:**
- Uses `GuardScopeFactory.projectScope()` for scope
- Test 1: `model classes do not contain business logic imports` -- files in `core.model..` should not import from domain/data/feature
- Test 2: `domain classes do not import Android framework` -- files in `core.domain..` should not import `android.*` or `androidx.*` (domain must be pure Kotlin)
- `assumeTrue` skips when packages have no files

**ModuleIsolationGuardTest.kt.template:**
- Uses `GuardScopeFactory.projectScope()` for scope
- Test 1: `feature modules do not import other feature modules` -- files in one `feature.X..` should not import `feature.Y..` (cross-feature isolation)
- `assumeTrue` if fewer than 2 feature packages exist
- AssertJ `withFailMessage` naming the offending import

All templates MUST:
- Use Unix line endings (LF, not CRLF) -- critical for sed on Windows/Git Bash
- Have `package __ROOT_PACKAGE__.konsist.guard` as package declaration
- Include KDoc header saying "Generated from AndroidCommonDoc guard-templates"

**setup/install-guard-tests.sh:**
- Shebang: `#!/usr/bin/env bash`, `set -euo pipefail`
- Header comment block with Usage, Options (matches existing install scripts)
- Options: `--package PACKAGE` (required -- consumer root package, e.g., `com.dawsync`), `--target DIR` (optional -- consumer project dir, defaults to cwd), `--dry-run`, `--force`, `--help`
- Colored logging: RED/GREEN/YELLOW/CYAN with `log_info`, `log_ok`, `log_warn`, `log_err` functions (same as install-claude-skills.sh)
- Script location resolution: `SCRIPT_DIR`, `COMMON_DOC`, `TEMPLATES_DIR="$COMMON_DOC/guard-templates"`
- Validate `--package` is provided (error with usage if missing)
- Validate `TEMPLATES_DIR` exists

Core logic:
1. Compute `PACKAGE_PATH` from package (dots to slashes): `${PACKAGE//\./\/}`
2. Set `TARGET_DIR="$TARGET/konsist-guard"`
3. Set `KOTLIN_DIR="$TARGET_DIR/src/test/kotlin/$PACKAGE_PATH/konsist/guard"`
4. Create directory structure: `mkdir -p "$KOTLIN_DIR"`
5. Copy + substitute `.kt.template` files: `for template in "$TEMPLATES_DIR"/*.kt.template; do sed "s/__ROOT_PACKAGE__/$PACKAGE/g" "$template" > "$KOTLIN_DIR/$(basename "$template" .template)"; done`
6. Copy + substitute `build.gradle.kts.template`: `sed "s/__ROOT_PACKAGE__/$PACKAGE/g" "$TEMPLATES_DIR/build.gradle.kts.template" > "$TARGET_DIR/build.gradle.kts"`
7. Modify consumer's `settings.gradle.kts`: check if `include(":konsist-guard")` already present (grep); if not, append `include(":konsist-guard")` at end of file. Idempotent.
8. Respect `--dry-run` (log but don't write) and `--force` (overwrite existing, default is skip)

Summary report at end (installed/skipped/errors counts).
