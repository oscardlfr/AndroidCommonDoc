## Summary

This phase is a comprehensive operational audit and modernization of the entire skill/agent/command infrastructure across three projects (L0 AndroidCommonDoc, L1 shared-kmp-libs, L2 DawSync) plus two subprojects (DawSyncWeb, SessionRecorder-VST3). The work covers seven interconnected areas: skill location consolidation (merging `.agents/skills/` into `skills/`), skill/agent modernization to current Claude Code 2026 patterns, GSD workflow integration verification, command-skill alignment, DawSyncWeb/subproject connectivity, sub-document pattern uniformity, and a full ecosystem quality gate with vault re-sync.

The current Claude Code platform has unified commands and skills -- files in `.claude/commands/*.md` and `.claude/skills/*/SKILL.md` both create `/slash-command` interfaces and work identically. Skills are recommended over commands because they support additional features: supporting files, subagent execution via `context: fork`, dynamic context injection via `!command` syntax, and `allowed-tools` restrictions. The key frontmatter fields are `name`, `description`, `disable-model-invocation`, `user-invocable`, `allowed-tools`, `model`, `context`, `agent`, and `hooks`. Subagents (`.claude/agents/`) are separate from skills and serve as specialized AI assistants with isolated contexts.

**Primary recommendation:** Consolidate `.agents/skills/` into `skills/`, modernize all skill/agent definitions to use current Claude Code frontmatter conventions, establish DawSyncWeb connectivity through automatic subdirectory skill discovery, verify GSD config correctness across all projects, fix all sub-document pattern violations, and run full quality gates with vault re-sync.

### Pattern 1: Skill Definition (Current Best Practice)

**What:** Modern Claude Code skill with full frontmatter
**When to use:** Every skill definition
**Example:**
```yaml
# Source: https://code.claude.com/docs/en/skills
---
name: test
description: "Run tests for a module with smart retry and error extraction. Use when asked to test, run tests, or check test results."
allowed-tools: Bash, Read, Grep, Glob
---

## Usage Examples
...

## Behavior
...

## Implementation
...
```

Key points from official docs:
- `description` is the most critical field -- Claude uses it to decide when to auto-load the skill
- `name` must be lowercase letters, numbers, and hyphens only (max 64 chars)
- `disable-model-invocation: true` for side-effect skills (deploy, commit)
- `user-invocable: false` for background knowledge Claude should know but users shouldn't invoke
- `allowed-tools` restricts tool access when skill is active
- `context: fork` runs skill in isolated subagent
- `$ARGUMENTS` for argument substitution, `!`command`` for dynamic context injection

### Pattern 2: Delegate Stub (L2 -> L0)

**What:** Lightweight file at L2 that points to L0 canonical definition
**When to use:** When an L2 project uses an L0 skill/command/agent
**Example:**
```yaml
---
delegate: true
canonical: "AndroidCommonDoc/skills/<name>/SKILL.md"
note: "Canonical version at L0. L2-specific overrides below this line."
---
```

**CRITICAL ISSUE:** Current DawSync skill delegates point to `AndroidCommonDoc/.agents/skills/<name>/SKILL.md` -- these paths will break after consolidation. Must be updated to `AndroidCommonDoc/skills/<name>/SKILL.md`.

### Pattern 3: Subagent Definition (Current Best Practice)

**What:** Subagent with focused system prompt, tool restrictions, and optional memory
**When to use:** Specialized AI assistants that need isolated context
**Example:**
```yaml
# Source: https://code.claude.com/docs/en/sub-agents
---
name: test-specialist
description: "Reviews test code for pattern compliance and identifies coverage gaps. Use when adding tests or reviewing test quality."
tools: Read, Grep, Glob, Bash
model: sonnet
memory: project
skills:
  - test                         # Preload skills into subagent context
---

Review test code for KMP project patterns:
...
```

Key features from official docs:
- `skills` field preloads skill content into subagent context (full content, not just reference)
- `memory` field (user/project/local) enables persistent cross-session learning
- `hooks` field for lifecycle events (PreToolUse, PostToolUse, Stop)
- `isolation: worktree` for git worktree isolation
- `background: true` for concurrent execution
- `maxTurns` limits agentic turns

### Pattern 4: Automatic Subdirectory Skill Discovery

**What:** Claude Code automatically discovers skills from nested `.claude/skills/` directories
**When to use:** Monorepo/subproject setups like DawSync + DawSyncWeb
**Source:** https://code.claude.com/docs/en/skills

When working with files in `packages/frontend/`, Claude Code also looks for skills in `packages/frontend/.claude/skills/`. This means DawSyncWeb at `DawSync/web/` can have `DawSync/web/.claude/skills/` and Claude will discover them automatically when editing web files.

### Anti-Patterns to Avoid

- **Dual skill locations:** Having skills in both `skills/` and `.agents/skills/` creates confusion about canonical location. Consolidate to ONE location.
- **Broken delegate paths:** Delegates that point to paths that no longer exist. Must verify all delegate paths resolve after any structural change.
- **Overly broad skill descriptions:** If description is too broad, Claude loads the skill when it shouldn't. Be specific about trigger conditions.
- **Skills without descriptions:** If `description` is omitted, Claude uses the first paragraph -- which may not clearly indicate when to use the skill.
- **Command files when skills would be better:** Commands are legacy. Skills support supporting files, subagent execution, tool restrictions. Prefer skills for new additions.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Skill discovery | Custom scanning logic | Claude Code automatic discovery from `.claude/skills/` dirs | Platform handles discovery, including nested subdirectories |
| Subagent delegation | Manual prompt composition | `context: fork` + `agent` field in skill frontmatter | Platform handles context isolation and execution |
| Skill preloading | Manual skill content injection | `skills` field in subagent frontmatter | Platform injects full content at startup |
| Permission control | Custom guard scripts (unless fine-grained) | `allowed-tools` in skill frontmatter + settings.json deny rules | Platform enforces tool restrictions natively |
| Dynamic data in skills | Pre-processing scripts | `!`command`` syntax in skill content | Platform runs commands and injects output before Claude sees the prompt |
| Copilot instructions | Manual sync of copilot files | `.github/copilot-instructions.md` + path-specific `*.instructions.md` | Platform auto-applies instructions |

## Common Pitfalls

### Pitfall 1: Delegate Path Breakage After Consolidation
**What goes wrong:** Moving skills from `.agents/skills/` to `skills/` breaks all DawSync delegates pointing to the old path.
**Why it happens:** Delegates use explicit `canonical:` paths. Path changes without updating all consumers silently break delegation.
**How to avoid:** Systematically find ALL files referencing `.agents/skills/` across all projects, update every reference, verify with grep.
**Warning signs:** DawSync skills show delegate stub content instead of actual skill content.

### Pitfall 2: Skill Description Budget Exhaustion
**What goes wrong:** Claude stops discovering skills because too many skill descriptions exceed the character budget (2% of context window, fallback 16,000 chars).
**Why it happens:** Adding 6 more skills to the 21 existing ones (total 27) increases description context load.
**How to avoid:** Keep descriptions concise (1-2 sentences). Use `disable-model-invocation: true` for skills that should only be manually invoked. Check `/context` for excluded skills warning.
**Warning signs:** Claude says "What skills are available?" and lists fewer than expected.

### Pitfall 3: Command-Skill Name Collision
**What goes wrong:** A skill and a command share the same name. The skill takes precedence, and the command version is silently ignored.
**Why it happens:** Skills at `.claude/skills/test/SKILL.md` and commands at `.claude/commands/test.md` both create `/test`.
**How to avoid:** This is actually the desired state -- skills should be the canonical definition. But be aware that command content is ignored when a matching skill exists.
**Warning signs:** Unexpected behavior when invoking a command that has a matching skill.

### Pitfall 4: Subproject .claude/ Isolation vs Inheritance
**What goes wrong:** Creating `.claude/` in DawSyncWeb doesn't automatically give it access to DawSync's parent skills/agents.
**Why it happens:** Claude Code scans skills from the working directory upward, but `.claude/` directories don't cascade -- each is independent.
**How to avoid:** Use `--add-dir` for cross-project skill access, or rely on automatic subdirectory discovery (Claude finds `web/.claude/skills/` when editing web files from the DawSync root).
**Warning signs:** Skills available when editing in DawSync root but not when editing DawSyncWeb files.

### Pitfall 5: GSD Config Branch Template Conflicts
**What goes wrong:** DawSync GSD config uses `feature/precloud-{slug}` branch template, which may conflict with worktree-based parallel development.
**Why it happens:** Branch template is per-project, and worktrees use the same git repository.
**How to avoid:** Verify branch template works correctly from worktree directories. Worktree agents should use their own feature branches per CLAUDE.md rules.
**Warning signs:** Branch creation fails or creates branches in unexpected locations from worktrees.

### Pitfall 6: shared-kmp-libs Commands Not Delegating to L0
**What goes wrong:** shared-kmp-libs has 9 direct commands (auto-cover, coverage-full, coverage, extract-errors, sync-versions, test-changed, test-full, test, verify-kmp) that are NOT delegates -- they're independent copies.
**Why it happens:** Phase 14 promoted DawSync commands to L0 with delegates but didn't update shared-kmp-libs.
**How to avoid:** Audit shared-kmp-libs commands, determine which should be delegates to L0, convert them.
**Warning signs:** shared-kmp-libs and L0 commands diverge over time because they're independent copies.

## Key Findings: Delegate Pattern Reality

**CRITICAL FINDING:** Claude Code does NOT resolve `canonical:` paths in delegate frontmatter. The delegate pattern used in this ecosystem is a human/documentation convention, not a platform-enforced mechanism. When Claude reads a delegate skill file, it sees the delegate stub content, NOT the canonical file's content.

**Implications:**
1. DawSync delegate skills (pointing to `.agents/skills/`) show stub text ("This skill has been promoted to L0...") -- NOT the actual skill instructions
2. For skills to work properly from DawSync, either:
   a. DawSync must use `--add-dir` to include AndroidCommonDoc's skills directory, OR
   b. Skills must be accessible through automatic subdirectory discovery (only works for nested dirs), OR
   c. Delegate files must contain the full skill content (defeating the purpose of delegation), OR
   d. DawSync skills must be full copies (not stubs) with a comment noting L0 as source of truth
3. The current delegate stubs are effectively broken -- they provide no useful instructions to Claude

**Recommendation:** Since Claude Code now supports `--add-dir` and skill discovery, the best approach is:
- L0 skills stay in `AndroidCommonDoc/skills/` (or `.claude/skills/`)
- DawSync configures `--add-dir` to AndroidCommonDoc path in settings, making L0 skills available
- Remove DawSync delegate stub skills entirely (they serve no purpose if Claude can discover L0 skills via --add-dir)
- Alternatively, DawSync delegate files become "override" files that only contain L2-specific additions
