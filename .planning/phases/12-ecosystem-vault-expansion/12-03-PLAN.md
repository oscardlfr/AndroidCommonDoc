**glob-expander.ts:**

Create a utility module that expands glob patterns against a root directory and returns matching file paths. Do NOT add npm dependencies -- use recursive directory walking with pattern matching.

```typescript
export interface GlobResult {
  /** Absolute path to the matched file */
  absolutePath: string;
  /** Path relative to the root directory (forward slashes) */
  relativePath: string;
}

/**
 * Expand multiple glob patterns against a root directory.
 * Returns deduplicated matching files, excluding files matching excludeGlobs.
 *
 * Supported glob syntax:
 * - `**` matches any directory depth
 * - `*` matches any characters in a filename segment
 * - Literal filenames (e.g., "CLAUDE.md") match exact name at root
 *
 * All paths normalized to forward slashes for cross-platform consistency.
 */
export async function expandGlobs(
  rootDir: string,
  includeGlobs: string[],
  excludeGlobs: string[],
): Promise<GlobResult[]>
```

Implementation approach:
1. Recursively walk the directory tree using `readdir` with `{withFileTypes: true}`
2. Skip directories matching common excludes (build, node_modules, .gradle, dist) for performance
3. For each found .md file, compute its relative path (forward slashes)
4. Test the relative path against each include glob using a `matchGlob` helper
5. Filter out paths matching any exclude glob
6. Deduplicate by absolute path
7. The `matchGlob` function should handle: literal filenames, `*` wildcards, `**` directory wildcards. Use simple pattern-to-regex conversion (not full minimatch). Escape special regex chars. `**` becomes `[^]*` (any chars including /). `*` becomes `[^/]*` (any chars except /).
8. Normalize all paths to forward slashes before matching.

**sub-project-detector.ts:**

Create a utility that auto-detects nested sub-projects within a project directory.

```typescript
import type { SubProjectConfig } from "./types.js";

/**
 * Signals that indicate a directory is an independent sub-project
 * (not just a build sub-module).
 *
 * Key insight from research Pitfall 7: Gradle sub-modules (core/, feature/)
 * are NOT sub-projects. Only directories with a DIFFERENT build system
 * or their own .git repo qualify.
 */
const CROSS_TECH_SIGNALS = ["CMakeLists.txt", "package.json", "Cargo.toml", "pyproject.toml"];
const INDEPENDENT_SIGNALS = [".git"];

export async function detectSubProjects(
  projectPath: string,
  maxDepth: number = 2,
  buildSystem?: string,  // Parent's build system (e.g., "gradle") to detect cross-tech
): Promise<SubProjectConfig[]>
```

Implementation approach:
1. Detect parent project's build system by checking for build.gradle.kts, package.json, Cargo.toml etc.
2. Scan immediate children (depth 1) and optionally grandchildren (depth 2, controlled by maxDepth)
3. Skip known non-project directories: build, .gradle, .git, node_modules, src, out, dist, .idea, .vscode
4. For each candidate directory, check for CROSS_TECH_SIGNALS that differ from the parent's build system OR INDEPENDENT_SIGNALS (.git submodule)
5. If a directory has CMakeLists.txt but parent is Gradle-based: it's a sub-project
6. If a directory has package.json but parent is Gradle-based: it's a sub-project
7. If a directory has .git: it's a sub-project (git submodule)
8. Build SubProjectConfig with name from directory basename, path as relative from project root
9. Do NOT treat Gradle sub-modules (directories with build.gradle.kts in a Gradle project) as sub-projects -- this is the key anti-pattern to avoid

Create `mcp-server/src/vault/version-catalog-parser.ts` that parses Gradle version catalog TOML files and generates a readable markdown reference page for the vault.

This implements the `features.versionCatalog` flag from ProjectConfig per CONTEXT.md locked decision: "Version catalog as reference doc: Configurable opt-in feature -- parse libs.versions.toml and generate a readable reference page."

```typescript
/**
 * Parse a Gradle version catalog (libs.versions.toml) and generate
 * a readable markdown reference page for the vault.
 *
 * @param tomlPath - Absolute path to libs.versions.toml
 * @returns Markdown content string, or null if file doesn't exist
 */
export async function parseVersionCatalog(tomlPath: string): Promise<string | null>
```

Implementation approach -- simple TOML section parser (do NOT add a TOML library dependency):

1. Read the file. If it doesn't exist, return null.
2. Parse TOML sections by detecting `[versions]`, `[libraries]`, `[bundles]`, `[plugins]` headers.
3. Within each section, parse `key = "value"` and `key = { module = "...", version.ref = "..." }` entries.
4. Generate markdown with frontmatter:
   ```markdown
   ---
   vault_type: reference
   vault_synced: {ISO date}
   ---
   # Version Catalog: {project name from path}

   > Auto-generated from `gradle/libs.versions.toml`. Source of truth is the TOML file.

   ## Versions

   | Key | Version |
   |-----|---------|
   | kotlin | 2.1.0 |
   ...

   ## Libraries

   | Alias | Module | Version |
   |-------|--------|---------|
   | kotlinx-coroutines | org.jetbrains.kotlinx:kotlinx-coroutines-core | 1.9.0 |
   ...

   ## Plugins

   | Alias | ID | Version |
   |-------|----|---------|
   ...

   ## Bundles

   | Name | Libraries |
   |------|-----------|
   ...
   ```

5. Handle TOML inline tables: `{ module = "group:artifact", version.ref = "key" }` -- extract module and resolve version.ref to the versions section.
6. Handle string values: `key = "value"` for simple version entries.
7. Handle multi-line arrays in bundles: `key = [\n  "lib1",\n  "lib2"\n]`.
8. Graceful handling: if parsing fails for a section, skip it with a warning comment in the output rather than throwing.

The parser does NOT need to handle the full TOML spec -- just the subset used by Gradle version catalogs. Keep it focused and maintainable.
