### File classification rules
- **Content-based classification** — frontmatter scope/targets determine primary domain, NOT filename prefix
- **Hub+sub-doc groups are absolute** — hub and all its sub-docs ALWAYS stay together in the same subdirectory, never split
- **New `category` frontmatter field** — explicit field (e.g., `category: testing`) added to all docs. Source of truth for subdirectory placement
- **Cross-domain docs** — file lives in primary domain subdir. Secondary domain's README includes a cross-reference link. Frontmatter scope lists both domains
- **Category vocabulary**: shared core + layer extras
  - L0 core: architecture, testing, error-handling, ui, gradle, offline-first, compose, resources, di, navigation, storage, guides
  - L1 additions: security, oauth, domain, firebase, foundation, io
  - L2 additions: business, product, legal, tech, references

### Cross-repo adaptation
- **L0 recommends, does not mandate** — doc-template.md documents recommended structure. Consumer projects adapt to their domain. Validation is opt-in
- **Opt-in validation** — validate-doc-structure checks ONLY projects where category frontmatter fields are present. Projects without category fields are skipped
- **Future L2 projects** (WakeTheCave, OmniSound) — same L0 core + L2 extensions pattern, recommended not enforced
- **L0 archive policy** — recommended baseline in doc-template.md: archive/ subdir in every project, archived files keep frontmatter with status: archived + archived_date, preserve original subdir structure inside archive/. Configurable/personalizable per project

### Auto-generated README.md
- **Every project** gets a docs/README.md auto-generated by the validate-doc-structure MCP tool
- **Content**: project description, subdirectory table (name, file count, description), classification rules legend
- **Optimized for AI context** and spec-driven development
- **Generated by MCP tool** — validate-doc-structure --generate-index mode

### Reference updates — atomic moves
- **Same phase, automated** — file moves AND all reference updates happen as a single atomic operation
- **All references updated**: cross-references in other .md files, versions-manifest.json paths, review-state.json paths, vault-config.json if needed, delegate files in DawSync (audited during execution — check if they reference local docs/ or L0 paths)
- **One commit per project** — AndroidCommonDoc, shared-kmp-libs, DawSync each get their own atomic commit. Order determined by planner
- **Legacy file renaming** — CONVENTION_PLUGINS.md → convention-plugins.md, API_EXPOSURE_PATTERN.md → api-exposure-pattern.md (in guides/, not archive/)

### MCP tooling updates
- **validate-doc-structure MCP tool** (new) — validates category field exists, file is in correct subdir, cross-references resolve. Cross-project: discovers all projects via project-discovery. --generate-index mode writes/updates docs/README.md
- **find-pattern category filter** — extend PatternMetadata with category field, add --category flag to find-pattern tool. Enables cross-layer queries by domain
- **MCP scanner verification** — run scanner after reorganization, verify docs:// URIs still resolve, find-pattern returns correct paths with new category field
- **Vault wikilinks** — keep slug-only format ([[testing-patterns]]). Obsidian resolves by unique filename regardless of subdirectory. No changes to wikilink-injector.ts

### Vault optimization
- **Mirror source subdirectory structure** — vault paths reflect source subdirs (L0-generic/testing/testing-patterns.md, not flat L0-generic/testing-patterns.md)
- **Refactor MOC generator** — update moc-generator.ts to group entries by category instead of flat listing. Category-grouped MOCs, not walls of links
- **Home.md redesign** — category-based navigation tree as single entry point
- **By Layer MOC** — show categories with key docs per category, not flat doc lists
- **By Project MOC** — show category counts per project, not every file
- **Exclude archive/** from vault — excludeGlobs: ["docs/archive/**"]. Active docs only in vault
- **Refactor existing code** — incremental updates to moc-generator.ts, transformer.ts, vault-writer.ts (not rewrite)

### /doc-reorganize skill (new)
- **Reusable skill** for any project — reads frontmatter category fields, moves files to matching subdirs, updates all cross-references, generates docs/README.md
- **Separate from vault sync** — /doc-reorganize handles file moves + references only. Vault sync is a manual separate step (/sync-vault)
- **Workflow**: /doc-reorganize → validate-doc-structure → /sync-vault (each step independent and verifiable)

<specifics>
## Specific Ideas

- "Everything should be optimized for context and spec-driven development with AI" — subdirs, README, frontmatter, vault all serve AI agent consumption
- "You can't force people to organize their docs the way you want" — L0 structure is recommended, not mandatory. Opt-in validation only
- "The structure should be configurable/personalizable" — archive policy and category vocabulary are customizable per project
- "By Project and By Layer MOCs are creating too much visual noise" — vault MOCs need category-grouped redesign, not flat link walls
- "Think about adding skills, agents, etc. for the whole doc management workflow (L0/L1/L2)" — broader doc management ecosystem deferred, but design current tools with extensibility in mind
- "Clean, solid, professional, enterprise, user-friendly approach" — quality bar for all implementations
- Model 5/5 AI-readiness docs (testing-patterns.md, viewmodel-state-patterns.md, compose-resources-patterns.md) remain the reference for frontmatter and content quality
