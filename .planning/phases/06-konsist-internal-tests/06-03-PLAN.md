    **ScriptParityTest.kt** in `com.androidcommondoc.konsist` package:

    Uses plain `java.io.File` operations + JUnit 5 + AssertJ.

    Tests:
    1. **`every sh script has a matching ps1 script`**
       - Scan `../scripts/sh/` for `.sh` files (exclude `lib/` subdirectory)
       - Scan `../scripts/ps1/` for `.ps1` files (exclude `lib/` subdirectory)
       - Canary: require shDir.exists() and ps1Dir.exists()
       - Canary: require shScripts.isNotEmpty()
       - Compute `missingPs1 = shScripts.nameWithoutExtension - ps1Scripts.nameWithoutExtension`
       - Assert missingPs1 is empty with message: "SH scripts without PS1 counterpart: $missingPs1. Create matching .ps1 files in scripts/ps1/"

    2. **`every ps1 script has a matching sh script`**
       - Reverse direction
       - Compute `missingSh = ps1Scripts.nameWithoutExtension - shScripts.nameWithoutExtension`
       - Assert missingSh is empty with message: "PS1 scripts without SH counterpart: $missingSh. Create matching .sh files in scripts/sh/"

    NOTE: Current codebase has some validate-phase03-* and validate-phase04-* .sh scripts without .ps1 counterparts. These tests WILL FAIL on current code, which is correct behavior -- the tests are catching a real parity gap. The executor should document this in the summary. If Phase 5 was supposed to clean these up (DEBT-04 removed orphaned scripts), check whether those validate-* scripts still exist. If they do, the test failure is legitimate and should be noted.

    ALTERNATIVE: If the validate-* scripts are intentionally sh-only (e.g., CI-only validation), the test could exclude known sh-only scripts via an explicit allowlist. But the cleaner approach is: if the scripts are orphans that should be deleted, let the test fail and document it as a known issue. Use the executor's judgment.
