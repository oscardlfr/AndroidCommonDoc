     - For generic KMP work: Load only L0 docs relevant to the task (use frontmatter scope/targets to filter)

     - For app work: Load L0 + L1 + L2 docs
     - Hub docs: Load the hub first (~100 lines), then load specific sub-docs based on need
     - Use frontmatter `scope` to filter: if working on testing, load docs where scope includes "testing"
     - Use frontmatter `targets` to filter: if working on iOS, load docs where targets includes "ios"

     - scope: Topic categories (array)
     - sources: Libraries/frameworks covered (array)
     - targets: Platforms (array)

     - parent: Hub doc slug (sub-docs only)
     - project: Project name (L1/L2 only)
     - monitor_urls: Upstream sources to track (versioning)

Create docs/storage-patterns.md (under 300 lines):

1. Frontmatter:
   - scope: [storage, data-persistence, architecture]
   - sources: [android-storage, ios-storage, sqldelight, datastore, mmkv]
   - targets: [android, desktop, ios, macos, jvm]
   - slug: storage-patterns
   - status: active
   - layer: L0
   - description: "Generic KMP storage patterns: platform models, encryption layers, key-value vs relational, expect/actual for storage"
   - monitor_urls: [DataStore releases, SQLDelight releases, MMKV releases]

2. Content sections:
   - **Overview**: Storage in KMP requires understanding platform differences (Android SharedPreferences/DataStore, iOS UserDefaults/Keychain, Desktop file-based) and choosing the right abstraction level.
   - **Platform Storage Models**:
     - Android: SharedPreferences (legacy), Jetpack DataStore (modern), EncryptedSharedPreferences, Room/SQLDelight
     - iOS/macOS: UserDefaults, Keychain (secure), Core Data, SQLDelight
     - Desktop/JVM: java.util.prefs, file-based, SQLDelight
     - Common patterns: all platforms support key-value and relational models
   - **Storage Category Decision Guide** (generic, not tied to specific libraries):
     - Key-value storage: When to use (settings, preferences, simple state)
     - Relational storage: When to use (structured data, queries, relationships)
     - Secure/encrypted storage: When to use (credentials, tokens, PII)
     - Caching: When to use (network response caching, computed values)
   - **Encryption Layer Patterns**:
     - Platform-specific crypto providers (Android Keystore, iOS Keychain, JVM KeyStore)
     - Transparent encryption wrappers (encrypt at storage boundary, not in business logic)
     - Key derivation and rotation patterns
     - SQLCipher for encrypted databases
   - **KMP expect/actual for Storage**:
     - Define storage interfaces in commonMain
     - Platform-specific implementations in androidMain/iosMain/desktopMain
     - Factory pattern for creating platform storage instances via DI
     - Testing: in-memory storage fakes for commonTest
   - **Migration Patterns**:
     - SharedPreferences to DataStore migration
     - Unencrypted to encrypted storage migration
     - Schema versioning for relational storage (SQLDelight migrations)
   - **Anti-patterns**:
     - Storing sensitive data in unencrypted key-value stores
     - Platform-specific storage APIs in commonMain (use expect/actual)
     - Synchronous I/O on main thread (use suspend functions)
     - Skipping schema migrations (leads to data loss)
   - **Related Patterns**: kmp-architecture.md (expect/actual), error-handling-patterns.md (storage errors). L1 reference: shared-kmp-libs storage-guide.md maps these concepts to specific modules.
