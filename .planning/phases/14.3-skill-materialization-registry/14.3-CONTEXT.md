<decisions>
## Implementation Decisions

### Registry schema & discovery (MATL-01)
- **Catalog everything**: skills (27), agents (11), and commands (32) in a single `skills/registry.json` — unified source of truth
- Each entry has `type` field: `skill`, `agent`, or `command`
- **Rich metadata per entry**: name, type, path, description, category (testing/quality/build/docs/...), tier (core/extended/web), hash (SHA-256 of content), dependencies (referenced scripts, other skills)
- **Versioning via content hash**: SHA-256 of file content — no manual version bumps needed. Downstream manifest stores hash, sync detects drift by comparing
- **Auto-generated**: a script/tool scans `skills/`, `.claude/agents/`, `.claude/commands/`, reads frontmatter, and generates `registry.json`. Runs as part of `/sync-l0` or standalone. Never out of sync with filesystem

### Manifest & sync behavior (MATL-02, MATL-03)
- **`l0-manifest.json`** in each L1/L2 project root declares: L0 source path, last-synced version (hash), selection rules, integrity checksums
- **Selection model**: Claude's discretion to design the cleanest approach — balance between auto-propagation of new L0 skills and explicit control. Enterprise-grade, spec-driven
- **Drift handling**: Show diff of what changed, then auto-apply. Like `git pull` — changes are visible but apply without interactive confirmation. Fast workflow for solo dev
- **Copy format**: Claude's discretion on whether materialized files get version headers (l0_source, l0_hash, synced_at) injected or use sidecar metadata. Choose the most solid, professional approach
- `/sync-l0` replaces `install-claude-skills.sh` entirely

### Adapter pipeline simplification (MATL-04)
- SKILL.md materializes directly without intermediate generated commands for Claude Code
- Copilot adapter retained for enterprise Microsoft support but simplified
- Claude's discretion on exact adapter simplification approach

### Migration strategy (MATL-05)
- **Big bang conversion**: convert everything in one pass across both shared-kmp-libs (9 delegates) and DawSync (32 cmd + 6 skill + 6 agent delegates)
- Create registry, create manifests, run `/sync-l0`, delete ALL old delegate stubs — clean cut
- **Delete entirely**: remove `install-claude-skills.sh`, `setup/templates/`, and all delegate-pattern infrastructure. No archiving, no fallbacks. `/sync-l0` is the single distribution mechanism
- **Verification**: Claude's discretion on the most solid automated verification approach — must confirm all manifest entries resolve, materialized files match L0 hashes, no orphaned delegates remain, L2-specific files untouched

### Skill validation tool (MATL-06)
- Parses all SKILL.md files, verifies frontmatter completeness
- Checks referenced scripts exist
- Validates registry is in sync with filesystem
- Safety net against broken skills after any change

### CLAUDE.md layering (MATL-07)
- Clean separation: `~/.claude/CLAUDE.md` = developer preferences (shared KMP rules), `{project}/CLAUDE.md` = project-specific rules only
- Eliminate duplication between layers — DawSync should go from 245 lines to much less
- Claude's discretion on the cleanest inheritance model (convention-based references, embedded subset, or other approach) — must be modern, 2026 best practices
- Claude's discretion on whether CLAUDE.md validation is part of `/sync-l0` or a separate concern — choose the most integrated, solid approach

### Vault naming & category consolidation (MATL-08)
- **Rename at source**: all ~120 DawSync UPPERCASE doc files renamed to `lowercase-kebab-case` directly in DawSync repo. Breaking change at source. Update all internal cross-references, frontmatter slugs, vault wikilinks
- **Category consolidation**: Claude's discretion to design the cleanest category model — shared core categories across all layers, with domain-specific extras allowed where semantically necessary. Target: ~8 unified categories, readable graph view
- Add naming validation to vault sync pipeline to prevent regression

### Final validation wave (MANDATORY)
- The LAST wave of the plan MUST be a comprehensive ecosystem validation:
  - Registry in sync with filesystem
  - All manifests valid, all materialized copies match L0 hashes
  - No orphaned delegates remain anywhere
  - Vault re-synced with renamed files and consolidated categories
  - CLAUDE.md layering correct across all projects (no duplication)
  - All existing validators pass (validate-doc-structure, script-parity, skill-script-alignment, template-sync)
  - READMEs updated to reflect new distribution model
  - Human-verified Obsidian graph view is clean and readable
- This is NOT a separate phase — it's the final wave of 14.3. Without it, the phase is not complete

### Claude's Discretion
