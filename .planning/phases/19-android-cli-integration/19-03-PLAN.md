# Plan 19-03: Skills Bridge + Module Lifecycle + Setup Wizard

**Phase:** 19 Android CLI Integration
**Wave:** 3
**Depends on:** 19-02 (uses `android-layout-diff` as reference pattern)
**Blast radius:** MEDIUM
**Requirements:** AC-CLI-03, AC-CLI-04, AC-CLI-05

## Goal

Google's skills discoverable by L0 without fork/redistribution (Option C). `android create` canonical path for new modules. `/setup` detects Android CLI install state.

## Scope

### Skills bridge (Option C — selective reference)
- New `skills/android-skills-consume/SKILL.md` with L0 frontmatter (`scope: l0`, `category: ecosystem`, `slug: android-skills-consume`)
- Content: documents how to install Google skills via `android skills add --agent=claude-code` and how L0 registry references them
- Extend `skills/sync-l0/SKILL.md` with new section "External skill sources" explaining that Google skills co-exist but are not synced from L0
- Extend `mcp-server/src/cli/generate-registry.ts` to emit `external_skills: []` array listing detected Google skills (read-only, informational)
- Consumer `l0-manifest.json` schema gets auto-populated entries in `selection.exclude_skills` for Google slugs that would collide with L0 slugs (collision guard)

### Module lifecycle
- Extend `setup/agent-templates/module-lifecycle.md` Step 1 "Naming" with "Option B: `android create`" alongside existing manual scaffold path
- New `docs/gradle/gradle-patterns-agp9.md` capturing templates that `android create` emits (L0 stays authoritative on opinions — convention plugins, build-logic, KMP layout)
- New `setup/create-module.sh` wrapping `android create` + post-processing that applies L0 convention plugin chain
- Sync edit to `.claude/agents/module-lifecycle.md`

### Setup hardening
- Extend `skills/setup/SKILL.md` Wizard with W9 "Android CLI detection" (mirror pattern of W3.5 Anthropic Skills at line 280-307):
  - Detect `android` CLI on PATH
  - Detect `$ANDROID_HOME`
  - On Windows hosts, detect `adb devices` has at least one authorized device (yellow-warn only, not blocker)
- `setup/setup-toolkit.sh` runs the detection; `/setup --verify-only` includes Android CLI in report

## Success Criteria

1. Running `/sync-l0` in DawSync after Google-published skill `android.screen-recorder` is installed locally does NOT copy it into DawSync's `.claude/skills/` nor conflict with L0 slugs
2. `setup/create-module.sh --name feature-foo --package com.dawsync.feature.foo` produces a module whose `build.gradle.kts` applies L0 convention plugins (`androidcommondoc.android.library` or equivalent) — not raw AGP config
3. `/setup --verify-only` on a Windows host without `android` CLI reports yellow-warn with install URL — does NOT fail the verify check
4. Byte-parity between `setup/agent-templates/module-lifecycle.md` and `.claude/agents/module-lifecycle.md`
5. `validate-skills` MCP tool confirms `skills/android-skills-consume/SKILL.md` has valid frontmatter with `scope: l0`, `category: ecosystem`
6. `cd mcp-server && npm test` green
7. `/pre-pr` green

## Risks & Mitigations

- **Collision guard false positives**: if a Google skill slug matches an L0 slug legitimately (unlikely), user must manually resolve in `l0-manifest.json`. Documented in `sync-l0` SKILL.
- **`create-module.sh` drift from L0 conventions**: add assertion in the script that runs `validate-kmp-packages` MCP tool on the created module before exit.
