# Roadmap: AndroidCommonDoc

## Overview

AndroidCommonDoc delivers a developer toolkit where patterns are correct by construction and AI agents spend tokens on logic, not boilerplate. The roadmap moves from stabilizing what exists (fixing parameter drift, completing pattern docs, defining canonical formats that generate both Claude Code commands and GitHub Copilot instructions) through building automated quality and enforcement infrastructure (freshness tracking, parity tests, Detekt rules, quality gates with Claude/Copilot template sync verification) to packaging everything for frictionless adoption (convention plugins, real-time enforcement for both Claude Code and Copilot).

**Cross-cutting concern: Claude Code + GitHub Copilot parity.** Every phase must produce artifacts that work equally for both tools. Canonical skill definitions generate Claude commands AND Copilot instructions. Quality gates verify both surfaces stay in sync. Distribution serves both tools, not just one.

## Phase Details

### Phase 1: Stabilize Foundation
**Goal**: Every pattern doc, script parameter, and skill definition is internally consistent and follows a single canonical format -- eliminating the multi-surface drift that compounds with every new feature, while ensuring Claude Code and GitHub Copilot receive identical guidance from the same source
**Depends on**: Nothing (first phase)
**Requirements**: PTRN-01, PTRN-02, SCRP-01, SCRP-02, TOOL-01, TOOL-02
**Success Criteria** (what must be TRUE):
  1. A developer reading any pattern doc (ViewModel, UI, testing, Gradle, offline-first, resources, compose resources) finds complete, layer-specific guidance with compilable code samples
  2. Running a validation script against pattern doc code samples produces zero version-mismatch or deprecated-API errors
  3. Every script parameter name is identical across its PS1 file, SH file, Claude command, and Copilot prompt (no --project-path vs --project-root drift)
  4. A script parameter manifest file exists that is the single source of truth for all script interfaces, and all surfaces reference it
  5. Running the canonical skill generator against a skill definition produces both a Claude Code command file and a GitHub Copilot instruction file with equivalent behavior -- and a new AI tool can be added by creating an adapter without modifying any existing Claude or Copilot files
**Plans:** 4/4 plans executed

Plans:

- [x] 01-02-PLAN.md -- Create parameter manifest and fix all naming drift across scripts/Makefile/Copilot
- [x] 01-03-PLAN.md -- Create canonical SKILL.md definitions for all 16 skills
- [x] 01-04-PLAN.md -- Build adapter scripts, regenerate all outputs, create AGENTS.md

### Phase 2: Quality Gates and Enforcement
**Goal**: Automated systems catch pattern violations, documentation staleness, script divergence, and cross-surface drift between Claude Code and GitHub Copilot -- so correctness and tool parity are verified continuously, not just when someone remembers to check
**Depends on**: Phase 1
**Requirements**: PTRN-03, SCRP-03, LINT-01, LINT-03, QUAL-01, QUAL-02, QUAL-03
**Success Criteria** (what must be TRUE):
  1. When a library version changes and a pattern doc references the old version, the freshness tracking system flags it automatically without human inspection
  2. Running the parity test suite against any PS1/SH script pair confirms identical output for the same inputs, and a divergence in behavior fails the test
  3. A consuming project that adds the Detekt rule JAR to `detektPlugins` gets build failures for non-sealed UiState, swallowed CancellationException, platform deps in ViewModels, missing WhileSubscribed timeout, and Channel-based UI events
  4. Compose-specific lint rules (mrmans0n 0.5.6) run alongside custom Detekt rules in the same build without conflicts
  5. Quality gate agents report a single pass/fail status covering script parity, skill-script alignment, Claude/Copilot template sync (verifying generated Claude commands and Copilot instructions are semantically equivalent), doc-code drift, and cross-surface parameter consistency -- with actual token cost data per skill
**Plans:** 3/3 plans executed

Plans:
- [x] 02-01-PLAN.md -- Custom Detekt rules module (5 AST-only architecture rules with TDD tests + Compose Rules integration)
- [x] 02-02-PLAN.md -- Freshness tracking (version manifest + doc-code-drift-detector agent + CI script pair)
- [x] 02-03-PLAN.md -- Quality gate agents and orchestrator (script-parity, skill-alignment, template-sync with cross-surface drift, unified orchestrator with token cost)

### Phase 3: Distribution and Adoption
**Goal**: A consuming project adopts the full toolkit (Detekt rules, Compose rules, pattern enforcement) with a single Gradle plugin application -- and gets real-time feedback during AI-assisted development via Claude Code hooks and GitHub Copilot instruction files that enforce the same patterns
**Depends on**: Phase 2
**Requirements**: LINT-02, TOOL-03
**Success Criteria** (what must be TRUE):
  1. A consuming project applies a single convention plugin in build-logic/ and gets all Detekt rules, Compose rules, and recommended configurations without manual detekt.yml editing
  2. Claude Code hooks fire during AI-assisted development in a consuming project and flag pattern violations in real-time before code is committed
  3. A consuming project that uses GitHub Copilot receives generated instruction files (from the convention plugin or toolkit setup) that enforce the same patterns as Claude Code hooks -- both tools produce equivalent guidance for the same codebase
**Plans:** 4 plans (3 executed + 1 gap closure)
