**Part B: Implement the doc-code-drift-detector agent body.**

The agent spec file `.claude/agents/doc-code-drift-detector.md` already exists with frontmatter and output format spec. Replace the agent body (everything below the frontmatter) with implementation instructions that the Claude agent can follow:

Keep the existing frontmatter (name, description, tools, model, memory) exactly as-is.

The implementation body must instruct the agent to:

1. **VERSIONS check:** Read `versions-manifest.json`. For each `docs/*.md` file, find the `> **Library Versions**:` line (or similar version header). Extract version numbers from that line. Compare each extracted version against the manifest:
   - Exact match: `[OK]`
   - Mismatch: `[STALE]` with both old and current version
   - Wildcard handling: For manifest entries with `x` (e.g., `1.7.x`), match if doc version matches `1.7.*` pattern (any patch)
   - Version not in manifest: skip (not tracked)

2. **VALIDATION GAPS check:** For each pattern doc, identify rule statements or checkboxes. Check if `/validate-patterns` or other commands actually validate them. Flag patterns with no validation backing.

3. **SCRIPT DRIFT check:** Compare script capabilities (read actual scripts in `scripts/ps1/` and `scripts/sh/`) against what docs describe. Flag scripts with features not in docs or docs describing non-existent features.

4. **ARCHITECTURE check:** Cross-reference `docs/kmp-architecture.md` and `docs/gradle-patterns.md` against validation scripts for accuracy.

5. **COMMAND REFS check:** Find all slash-command references in docs (e.g., `/validate-patterns`, `/test`). Verify each exists in `.claude/commands/`. Flag renamed or removed commands.

6. **Output:** Produce the structured report matching the existing output format spec exactly (VALIDATION GAPS, SCRIPT DRIFT, VERSIONS, ARCHITECTURE, COMMAND REFS sections with [GAP]/[OK]/[DRIFT]/[STALE]/[BROKEN] prefixes and OVERALL summary line).

The agent must use Read, Grep, and Glob tools (as declared in frontmatter). No Bash execution needed -- this is static file analysis.

<task type="auto">
  <name>Task 2: Create CI freshness check script pair (PS1/SH)</name>

  </files>
  <action>
Create a lightweight CI script pair that performs ONLY the version freshness check (the VERSIONS section of the full agent). This is the fast CI path -- the full agent does comprehensive drift detection, but CI only needs version staleness.

Both scripts must:
1. Read `versions-manifest.json` from repo root
2. Scan all `docs/*.md` files for version references in headers (the `> **Library Versions**:` line)
3. Extract library name + version pairs from those lines
4. Compare each against the manifest
5. Print structured output:
   ```
   Doc Freshness Check

   [OK] docs/viewmodel-state-patterns.md -- all versions current
   [STALE] docs/gradle-patterns.md -- AGP: found 8.x, expected 9.0.0

   Result: PASS (0 stale) / FAIL (N stale)
   ```
6. Exit code: 0 if all versions match, 1 if any stale version found

**PowerShell script (`check-doc-freshness.ps1`):**
- Use `param()` block with `[string]$ProjectRoot = "."` parameter (following established naming from params.json)
- Parse versions-manifest.json with `ConvertFrom-Json`
- Use `Select-String` or `Get-Content` to find version lines in docs
- Regex to extract version numbers: match patterns like `Kotlin 2.3.10`, `AGP 9.0.0`, `Compose 1.7.x`
- Wildcard handling: if manifest version contains 'x', compare only up to the wildcard position

**Bash script (`check-doc-freshness.sh`):**
- Use `getopts` for `--project-root` parameter (matching PS1's ProjectRoot)
- Parse versions-manifest.json with `python3 -c "import json,sys; ..."` (following the adapter pattern from Phase 1 -- python3 for JSON parsing in Bash)
- Use `grep` to find version lines in docs
- Same regex extraction and comparison logic as PS1
- Same output format and exit codes

**Parity requirements** (these scripts will themselves be subjects for the script-parity-validator):
- Same parameter names (ProjectRoot/--project-root per existing convention)
- Same output format (identical [OK]/[STALE] lines)
- Same exit codes (0=pass, 1=fail)
